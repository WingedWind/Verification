# Лабораторная работа №2
## Верификация модели автомата выдачи корма для белок в SPIN

### Тема работы
В данной лабораторной работе рассматривается система **"Автомат выдачи корма для белок"** из предложенного списка вариантов.

### Цель работы
Построить модель параллельной системы на языке **Promela (SPIN)**, сформулировать свойства на языке **LTL** и провести их верификацию.

Согласно требованиям лабораторной работы №2, необходимо:
- взять систему из лабораторной работы №1;
- записать её спецификацию на входном языке SPIN;
- сформулировать 3 свойства системы на языке темпоральной логики LTL;
- провести верификацию этих свойств в SPIN;
- сравнить методы моделирования и верификации в nuXmv и SPIN.

Требования к системе и свойствам остаются такими же, как и в лабораторной работе №1.

---

## Описание модели

Во второй лабораторной работе используется та же система, что и в первой: автомат выдачи корма для белок.

Модель построена на языке **Promela** и состоит из нескольких параллельных процессов:

1. `Environment` — внешняя среда.  
   Моделирует появление белки у автомата и запрос на пополнение корма.

2. `PresenceSensor` — датчик присутствия.  
   Передаёт сигнал о наличии белки в управляющий модуль.

3. `Controller` — управляющий алгоритм.  
   Имеет состояния `idle`, `dispense`, `wait_leave` и принимает решение о выдаче корма.

4. `Actuators` — исполнительные механизмы.  
   Управляют мотором выдачи корма и сигнальной лампой.

5. `Storage` — модуль хранения корма.  
   Уменьшает количество порций при выдаче и восстанавливает запас при пополнении.

Таким образом, модель соответствует требованиям лабораторной работы, поскольку содержит:
- процесс среды;
- процесс датчика;
- процесс управляющего алгоритма;
- процессы актуаторов и индикации;
- бесконечное функционирование за счёт недетерминированных внешних событий.

---

## Особенности модели в SPIN

В отличие от nuXmv, где модель задаётся декларативно через переходы состояний, в SPIN система задаётся как набор взаимодействующих процессов на языке Promela.

Чтобы сделать поведение модели более понятным и устойчивым при проверке LTL-свойств, используется переменная `turn`. Она организует циклическое выполнение процессов в фиксированном порядке:

- `0` — `Environment`
- `1` — `PresenceSensor`
- `2` — `Controller`
- `3` — `Actuators`
- `4` — `Storage`

Такой подход позволяет:
- явно разделить функции среды, датчика, контроллера, актуаторов и хранилища;
- сохранить параллельную структуру системы;
- упростить интерпретацию переходов;
- удобнее формулировать свойства в LTL.

## Основные переменные модели

### Переменные среды и датчика
- `squirrel_here` — белка находится у автомата;
- `sensor_present` — датчик обнаружил белку;
- `absence_timer` — таймер отсутствия белки;
- `refill_request` — запрос на пополнение корма;
- `refill_timer` — таймер до пополнения.

### Переменные контроллера и актуаторов
- `controller_state` — состояние контроллера (`idle`, `dispense`, `wait_leave`);
- `dispense_cmd` — команда на выдачу корма;
- `motor_on` — мотор включён;
- `lamp_on` — сигнальная лампа включена.

### Переменные хранилища
- `food_count` — количество порций корма;
- `MAX_FOOD` — максимальная ёмкость автомата.

---

## Логика работы системы

Работа модели организована следующим образом:

1. Среда моделирует появление и исчезновение белки.
2. Если белки долго нет, через некоторое время модель гарантирует её повторное появление.
3. Когда корм заканчивается, через несколько шагов формируется запрос на пополнение.
4. Датчик считывает присутствие белки и передаёт сигнал контроллеру.
5. Контроллер при наличии белки и корма переходит в состояние `dispense`.
6. Актуаторы включают мотор выдачи и обновляют состояние лампы.
7. Хранилище уменьшает запас корма при выдаче и восстанавливает полный запас после пополнения.

## Проверяемые свойства

Для модели были заданы 4 свойства на языке LTL.

### Свойство 1
```promela
ltl p0 { []<> (squirrel_here) }
```

**белка появляется бесконечно часто.**

Это свойство показывает, что внешняя среда остаётся активной, а система не переходит в состояние, в котором белки больше никогда не появляются.

---

### Свойство 2
```promela
ltl p1 { [] ( ((turn == 2) && sensor_present && (food_count > 0) && (controller_state == idle))
              -> <> (controller_state == dispense) ) }
```

**если на шаге контроллера белка обнаружена, корм есть и контроллер находится в состоянии ожидания, то в будущем контроллер перейдёт в состояние выдачи корма.**

Это свойство отражает корректную реакцию управляющего алгоритма на появление белки.

---

### Свойство 3
```promela
ltl p2 { [] ( refill_request -> (food_count == 0) ) }
```

**запрос на пополнение корма может возникнуть только тогда, когда корм действительно закончился.**

Это свойство относится к свойствам безопасности и проверяет корректность логики внешней среды.

---

### Свойство 4
```promela
ltl p3 { [] ( (food_count == 0) -> <> (food_count == MAX_FOOD) ) }
```

**если корм закончился, то в будущем автомат снова станет полностью заполненным.**

Это свойство относится к живости и описывает восстановление ресурса.

## Сравнение nuXmv и SPIN

### Сходство
- В обеих лабораторных работах используется одна и та же система.
- В обеих моделях выделены среда, датчик, контроллер, актуаторы и хранилище.
- В обеих работах проверяются свойства живости и безопасности.

### Отличия
1. **Стиль моделирования**  
   В nuXmv модель задаётся декларативно через переменные и переходы `next(...)`.  
   В SPIN модель задаётся как исполняемая программа на языке Promela.

2. **Представление параллелизма**  
   В nuXmv переход описывает изменение всей системы сразу.  
   В SPIN взаимодействие процессов моделируется явно, поэтому важен порядок выполнения и межпроцессное чередование.

3. **Формулировка свойств**  
   В nuXmv использовались формулы с операторами `G`, `F`, `X`.  
   В SPIN аналогичные свойства записываются через `[]` и `<>`.

4. **Отладка**  
   В nuXmv удобно анализировать модель как систему переходов.  
   В SPIN удобно исследовать поведение процессов и трассы выполнения.

### Вывод сравнения
nuXmv удобнее для компактного описания переходной системы и логического анализа состояний. SPIN удобнее для моделирования взаимодействующих процессов и пошагового исследования сценариев выполнения. Для данной задачи оба инструмента подходят, но SPIN требует более явного описания порядка действий процессов.
