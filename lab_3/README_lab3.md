# Лабораторная работа №3
## Верификация модели автомата выдачи корма для белок в TLA+ Toolbox

### Что было не так в исходных файлах

Изначальная версия для лабораторной работы №3 не полностью соответствовала требованиям задания.

Основные проблемы:
- модель не совпадала по смыслу с системой из лабораторной работы №1;
- отсутствовал явный учёт запаса корма и пополнения автомата;
- в `.cfg` не были подключены `SPECIFICATION`, `INVARIANT` и проверяемые свойства, поэтому TLC фактически мог не проверять нужные формулы;
- было задано слишком мало корректно оформленных свойств;
- в спецификации не хватало fairness, из-за чего TLC мог строить бесконечную stuttering-трассу и нарушать свойства живости;
- файл `hs_err_pid...log` является журналом аварийного завершения Java/TLA+ Toolbox и не относится к модели.

В исправленной версии модель приведена к той же логике, что использовалась в лабораторных работах 1 и 2: среда, датчик, контроллер, актуаторы и хранилище корма.

## Описание модели

Модель записана на языке TLA+ и описывает параллельную систему «Автомат выдачи корма для белок».

Используются следующие логические компоненты:
- среда (`EnvStep`);
- датчик присутствия (`SensorStep`);
- контроллер (`ControllerStep`);
- актуаторы (`ActuatorStep`);
- хранилище корма (`StorageStep`).

Для удобства выполнения система организована через переменную `phase`, которая задаёт последовательность шага:
`env -> sensor -> ctrl -> act -> store -> env`.

Это позволяет сохранить разбиение на компоненты и при этом сделать модель удобной для TLC.

### Основные переменные
- `squirrelHere` — белка находится у автомата;
- `sensorPresent` — датчик видит белку;
- `controllerState` — состояние контроллера (`idle`, `dispense`, `wait_leave`);
- `dispenseCmd` — команда на выдачу корма;
- `motorOn` — мотор включён;
- `lampOn` — сигнальная лампа включена;
- `foodCount` — количество порций корма;
- `absenceTimer` — таймер отсутствия белки;
- `refillRequest` — запрос на пополнение;
- `refillTimer` — таймер до пополнения;
- `phase` — текущий этап цикла работы системы.

---

## Проверяемые свойства

В исправленной версии используются четыре свойства.

### P1
`[]<>(squirrelHere = TRUE)`

Белка появляется бесконечно часто. Это свойство показывает, что внешняя среда остаётся активной.

### P2
`[]((phase = "ctrl" /\ sensorPresent /\ foodCount > 0 /\ controllerState = "idle") => <>(controllerState = "dispense"))`

Если на шаге контроллера обнаружена белка, есть корм и контроллер находится в состоянии ожидания, то в будущем система перейдёт к выдаче корма.

### P3
`[](refillRequest => foodCount = 0)`

Запрос на пополнение может быть установлен только тогда, когда корм закончился.

### P4
`[]((foodCount = 0) => <>(foodCount = MAX_FOOD))`

Если корм закончился, то в будущем запас будет восстановлен.

## Что проверять в TLC

Файл конфигурации `squirrel_feeder.cfg` содержит:
- константы;
- спецификацию `Spec`;
- инвариант `TypeOK`;
- свойства `P1`, `P2`, `P3`, `P4`.

В спецификации используется fairness:
`Spec == Init /\ [][Next]_Vars /\ WF_Vars(Next)`

Это нужно для корректной проверки свойств живости и исключения бесконечного stuttering-поведения.

Таким образом, модель соответствует требованиям лабораторной работы №3: используется система из лабораторной работы №1, заданы свойства на языке TLA+, и они проверяются в TLC.

## Команда запуска

Проверку нужно запускать так:

```bash
java -cp tla2tools.jar tlc2.TLC -config squirrel_feeder.cfg squirrel_feeder.tla
```

Если всё настроено правильно, TLC должен завершить проверку сообщением:

```text
Model checking completed. No error has been found.
```

## Что написать в сравнении nuXmv, SPIN и TLC

### nuXmv
В nuXmv модель задаётся декларативно через переменные и функции перехода `next(...)`. Это удобно для компактного описания системы переходов.

### SPIN
В SPIN модель задаётся как исполняемая программа на Promela с набором процессов. Это удобно для анализа поведения системы как взаимодействующих процессов.

### TLA+ / TLC
В TLA+ система описывается через состояния и действия. TLC хорошо подходит для проверки свойств, когда важно явно отделить начальное состояние, отношение перехода и темпоральные свойства.

### Общий вывод
- nuXmv удобен для компактных переходных систем;
- SPIN удобен для моделирования алгоритма как программы;
- TLA+ удобен для формального описания состояния и действий на более абстрактном уровне.
